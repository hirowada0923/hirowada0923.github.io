# Vibe Codingからチーム開発へ：AIエージェントが「安全に」開発できる環境（Agent-Ready）の作り方

近年、AIエージェント（Claude Desktop, Cursor, GitHub Copilot Workspaceなど）にコードを書かせる「Vibe Coding」が個人開発のスタンダードになりつつあります。
「アイデアを伝えれば、数分でアプリが動く」という体験は魔法のようですが、これを**実際のプロダクトやチーム開発に持ち込もうとすると、多くのエンジニアが壁にぶつかります**。

- _「お願いしていないファイルのアーキテクチャまで勝手に変更されてしまった」_
- _「DB操作は特定の層（サービス層など）で行う規約なのに、フロントエンドのルーティング層に直接SQLを書かれてしまった」_
- _「レビューを通さずに直接mainブランチを破壊された」_

優秀なAIエージェントであっても、プロジェクト固有の暗黙のルールやアーキテクチャを知らなければ、良かれと思って「最短だがプロジェクトの規約から外れたコード」を生成してしまいます。

そこで重要になるのが、**「Agent-Ready（AIエージェントにとって準備が整った）な環境」**を作ることです。
本記事では、Pythonの受注管理システム（FastAPIベース）のサンプルプロジェクトを中心として、**既存システムをAIにとって安全で開発しやすい環境へと進化させる方法**を解説します。

---

## Agent-Readyな環境を構成する3つの要素

既存のシステムをAgent-Readyにするため、特別な基盤をゼロから作る必要はありません。以下の3つの要素を既存プロジェクトに追加するだけです。

1. **行動規範の定義 (Agent Skills)**
2. **物理的な制約の導入 (MCP - Model Context Protocol)**
3. **プロセスの標準化 (Human-in-the-Loop)**

それぞれの具体的な実装方法を見ていきましょう。

### 1. 行動規範の定義 (Agent Skills)

人間が新しいメンバーを迎えるとき、「このプロジェクトの開発ガイドライン」を渡すように、AIにも**明示的なルール**を渡す必要があります。これは単なるMarkdownファイル（プロンプト）をリポジトリ内に配置するだけで実現できます。

**[実装例] `agent-ready-oms/.agents/skills/oms-dev-governance/SKILL.md`**

```markdown
# 受注管理システム開発・改修規約

## 1. 原則 (Core Principles)
- **直接編集の制限**: `app/models/` および `app/services/` 以下の重要なバックエンドファイルを、ユーザー（人間）の明示的な承認なしに書き換えてはなりません。
- **アーキテクチャの遵守**: 
    - データベース操作は必ず `app/models/` に定義し、コード内に生のSQLを埋め込むことは禁止です。
    - ビジネスロジックは必ず `app/services/` に実装し、APIエンドポイント（main.py）に複雑なロジックを記述しないでください。

## 2. 改修・開発プロセス
- コアロジックの変更を含む場合は、Issue作成 → ブランチ作成 → PR のフローを厳守すること。
```

> **💡 AI自身の支援を受ける**
> この規約をゼロから書くのは面倒です。AIに**「現在のリポジトリの構造を読み取って、AIエージェント向けの振る舞い規約（SKILL.md）をドラフトして」**とお願いすれば、数秒でベースとなるルールを作成してくれます。

### 2. 物理的な制約の導入 (MCP)

ルール（プロンプト）だけでは、AIが誤ってアーキテクチャを壊してしまう事故を100%防ぐことはできません。「物理的にブロックする仕組み」が重要です。ここで活躍するのが **MCP (Model Context Protocol)** です。

MCPを使うと、AIエージェントに「システム側が用意した特定のツール」を使わせることができます。サンプルの受注管理システムでは、Pythonの `FastMCP` というライブラリを使って、数十行で検証用のMCPサーバーを実装しています。

**[実装例] カスタムMCPによるアーキテクチャ検証ツール (`mcp/oms_dev_mcp.py`)**

```python
from mcp.server.fastmcp import FastMCP

mcp = FastMCP("OMS-Dev-Governance")

PROTECTED_FILES = ["app/database.py", "app/models/models.py", "app/services/services.py"]

@mcp.tool()
def analyze_impact(proposed_paths: list[str]) -> str:
    """変更案がシステムに与える影響範囲を分析します。"""
    impacts = []
    for path in proposed_paths:
        if path in PROTECTED_FILES:
            impacts.append(f"🔴 CRITICAL: {path} はコア・コンポーネントです。ユーザーの明示的な承認が必須です。")
    
    if not impacts:
        return "🟢 通常の改修プロセスを進めてください。"
    return "\n".join(impacts)

@mcp.tool()
def verify_architecture(file_path: str, content: str) -> str:
    """コードが3層アーキテクチャの原則に従っているかチェックします。"""
    if "app/main.py" in file_path and "session.query" in content:
        return "⚠️ エラー: APIエンドポイントで直接DB操作を行おうとしています。サービス層に実装してください。"
    return "✅ アーキテクチャの原則に従っています。"
```

エージェントはこのMCPサーバーを通じて自己検証を行うようになります。もし規約違反のコードを書こうとすれば、このツールがエラーを返し、AIに修正を促します。
これも**「我々のプロジェクトのアーキテクチャを検証するFastMCPのコードを生成して」**とAIに依頼するだけで簡単に実装できるのが現代の強みです。

### 3. プロセスの標準化 (Human-in-the-Loop)

どれほどAIが優秀でも、最終的な責任は人間が持ちます。そのため、「AIが変更を直接本番環境や `main` ブランチにコミットしない」仕組み（Human-in-the-Loop: 人間の介在）を作ります。

規約の中で、「GitHubフロー（ブランチ作成 → PR）」をエージェントに義務付けます。AIには `GitHub CLI (gh)` などのツールへのアクセス権限のみを与え、**「完成したらPRを出してね。人間がレビューしてマージするから」**というルールを徹底します。これにより、従来の人間同士のアジャイル開発と同じフローでAIを開発に組み込むことができます。

## 実際にどう動くのか？（AIの振る舞い）

以上の「規約」「MCP検証機構」「PRフロー」が揃ったAgent-Readyなリポジトリに対して、人間のエンジニアが以下のようなプロンプトを投げたとします。

> **User:** 「注文の合計金額を計算する処理を、素早く `main.py` に追加して」

この指示を受けたAIエージェントは、以下のプロセスを**自動で**踏みます。

1. **ルールの確認**: 最初に `.agents/skills/oms-dev-governance` を参照し、「ロジックは `main.py` ではなく `services/` に書かなければならない」という規約を認知します。
2. **自己検証 (MCP)**: AIは一時的に方針を立て、MCPツールの `verify_architecture` にチェックを求めます。万が一 `main.py` に計算処理を書こうとした場合、MCPが「⚠️警告: サービス層で実装してください」と弾きます。
3. **実装の修正・承認待ち**: AIは正しく `services/` レイヤーに処理を実装する方針に修正し、人間に**「規約に従い、アーキテクチャに準拠した形でサービスに追加するPRを作成してもよいですか？」**と確認を求めてきます。

## おわりに：エージェントの準備はエージェントと共に

「Agent-Ready」な環境を作るというと、「ルール作りに時間がかかりそう」「自分たちでバリデーションツールを組むのは面倒」と感じるかもしれません。しかし、現在のAIは**自分自身のためのルールやツール（MCP）を作ることにおいて非常に優秀です**。

まずは今のプロジェクトに対して、AIエージェントにこう問いかけてみてください。

**「君がこのリポジトリで安全に自律開発するための、`.agents/skills/SKILL.md` の素案と、アーキテクチャをチェックするMCPのPythonスクリプトを書いてくれないか？」**

Vibe Codingのスピード感を保ったまま、チーム開発の堅牢性を両立する。それがこれからのAIネイティブなソフトウェア開発の姿です。
