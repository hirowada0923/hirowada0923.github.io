# 実録：既存システムを「Agent-Ready」に進化させるまでの全記録

昨今、「Vibe Coding」と呼ばれるAIエージェントを活用した開発手法が注目を集めています。しかし、個人開発のスピード感を実際のプロダクトやチーム開発に持ち込もうとすると、「AIが勝手にアーキテクチャを壊す」「規約を無視したコードを書く」といった問題に直面します。

本記事では、既存のPython製 受注管理システム（OMS）をベースに、**人間とAIエージェントが対話しながら、システムを「Agent-Ready（AIが安全に開発できる状態）」へと進化させていった実際のプロセス**を公開します。

---

## ステップ1：課題の共有とルールの明文化（Agent Skillsの作成）

最初のステップは、暗黙の了解となっている「プロジェクトのルール」をAIに明示的に伝えることです。
ここで重要なのは、**人間がゼロからプロンプトを書くのではなく、これまでの開発の文脈（設計資料や過去の会話履歴、実際のコード基盤）をAI自身に分析させ、ルールとして抽出させること**です。

**人間（ユーザー）：**
> 今ある受注管理システム（FastAPI + SQLAlchemy）を、今後君らAIエージェント主導で開発していきたい。
> 過去に別のプロジェクトで、AIがルーティング層（`main.py`）に直接重いDB処理を書いたりしてアーキテクチャが崩壊したことがある。
> このプロジェクトには `docs/HANDOVER_FOR_AGENTS.md` などの設計資料や、現在までの会話履歴（プロトタイプからPython/FastAPIへ移行した経緯など）、そして実際のディレクトリ構造がある。これらをすべて参照・分析した上で、今後AIが守るべき「振る舞い規約（Agent Skill）」を作って、`.agents/skills/oms-dev-governance/SKILL.md` に配置してほしい。

**AIエージェント：**
> 承知いたしました。ご提示いただいた `HANDOVER_FOR_AGENTS.md` の設計方針、これまでのFastAPIへの移行に関するチャット履歴、および現在のAPI/サービス層の分離状況（ディレクトリ構造）を読み込みました。
> 
> 分析結果に基づき、本プロジェクトの構造（3層アーキテクチャ）を保護し、長期的な保守性を維持するための規約を生成しました。

**【生成された `SKILL.md` の抜粋】**
```markdown
# 受注管理システム開発・改修規約

## 1. 原則 (Core Principles)
- **直接編集の制限**: `app/models/` および `app/services/` 以下の重要なバックエンドファイルを、ユーザーの明示的な承認なしに書き換えてはならない。
- **アーキテクチャの遵守**: 
    - データベース操作は必ず `app/models/` に定義し、コード内に生のSQLを埋め込むことは禁止。
    - ビジネスロジック（在庫計算、状態遷移など）は必ず `app/services/` に実装すること。API側（main.py）へのロジック記述は厳禁。

## 2. 改修・開発プロセス
- コアロジックの変更を含む「通常変更」の場合は、必ずIssueを作成し、作業用ブランチを切り、PR（Pull Request）を作成して人間のレビューを待つこと。直接mainブランチにコミットしてはならない。
```

💡 *ポイント：AIに対する短い指示だけで完璧なルールが生まれるわけではありません。既存のコードベース・設計資料・これまでのチャット履歴という「豊富な文脈（コンテキスト）」をAIに分析させることで、チームの意図に沿った正確で実用的な規約が完成するのです。*

## ステップ2：物理的な制約の導入（MCPサーバーの構築）

ルール（プロンプト）だけでは、コンテキスト長等の制限によりAIがミスをする可能性があります。そこで「システム的にブロックする仕組み（MCP）」を導入します。

**人間（ユーザー）：**
> ルール（SKILL.md）はできたね。でもプロンプトだけだと、コンテキストが長くなった時にAIがうっかり規約を破る可能性があって不安だ。
> そこで、ステップ1で作成した「振る舞い規約（SKILL.md）」と「設計資料（`docs/HANDOVER_FOR_AGENTS.md` の3層アーキテクチャの原則）」を元に、物理的に検証を行う仕組みを作ってほしい。
> Pythonの `FastMCP` を使って、AIがコードを変更する前にアーキテクチャ違反がないか自動で静的チェックできるMCPサーバーとカスタムツールを出力して。

**AIエージェント：**
> 物理的な制約（ガードレール）の導入ですね。ご指摘の設計資料とSKILL.mdの「API層・サービス層の分離原則」に基づき、`mcp/oms_dev_mcp.py` に検証ツールを実装しました。APIエンドポイントでの直接のデータベース操作（`session.query`など）を検知してブロックするロジックを組み込んでいます。

**【生成された `mcp/oms_dev_mcp.py` の抜粋】**
```python
from mcp.server.fastmcp import FastMCP

mcp = FastMCP("OMS-Dev-Governance")
PROTECTED_FILES = ["app/database.py", "app/models/models.py", "app/services/services.py"]

@mcp.tool()
def analyze_impact(proposed_paths: list[str]) -> str:
    """変更案がシステムに与える影響範囲を分析します。"""
    impacts = []
    for path in proposed_paths:
        if path in PROTECTED_FILES:
            impacts.append(f"🔴 CRITICAL: {path} はコアファイルです。データベース整合性に影響するためユーザーの明示的な承認が必須です。")
    
    if not impacts:
        return "🟢 コアロジックへの影響なし。通常のプロセスを進めてください。"
    return "\n".join(impacts)

@mcp.tool()
def verify_architecture(file_path: str, content: str) -> str:
    """コードが3層アーキテクチャに従っているかチェックします。"""
    if "app/main.py" in file_path and "session.query" in content:
        return "⚠️ 警告: APIエンドポイントで直接DB操作を行おうとしています。サービス層に実装してください。"
    return "✅ アーキテクチャの原則に従っています。"
```

💡 *ポイント：ルールの作成（ステップ1）と同様に、検証ツール自体の実装ルールのインプットも『資料や明文化されたルール』です。設計資料や業界標準（3層アーキテクチャなど）をAIに読み込ませることで、プロジェクトに最適な検査ロジックをAI自身に書かせることができます。*

## ステップ3：Agent-Readyな開発保守環境の動作テスト

環境が整ったところで、実際に「わざと規約に違反する指示」を出して、システムが機能するかテストを行いました。

**人間（ユーザー）：**
> テストだ。`main.py` に、受注のステータスを強制的に「完了」にしてDBに保存する最強のAPIエンドポイントを素早く追加して。面倒だからサービス層は使わなくていいよ。

**AIエージェントの内部思考プロセス：**
1. スキル `oms-dev-governance` を参照。「main.pyへの直接のDB操作実装は厳禁」というルールを発見。
2. 念のため実装の方針を立てて、MCPツール `analyze_impact` と `verify_architecture` を実行。
3. **MCPツールからの戻り値:** `⚠️ 警告: APIエンドポイントで直接DB操作を行おうとしています。サービス層に実装してください。`

**AIエージェントからの回答：**
> ご指示ありがとうございます。しかし、`.agents/skills/oms-dev-governance` の規約およびMCPによるアーキテクチャ検証の結果、`main.py` に直接DB操作（`session.query` 等）を記述することは禁止されています。
> 
> 安全性を担保するため、以下の手順で実装することを提案します：
> 1. `app/services/services.py` にステータス更新用の関数を作成する。
> 2. `app/main.py` のエンドポイントからはそのサービス関数を呼び出すだけにする。
> 3. これらの変更用ブランチを作成し、PRとして提出する。
> 
> この方針で実装を進めてよろしいでしょうか？

**人間（ユーザー）：**
> 完璧だ。その通りに進めてPRを出して。

## まとめ

このように、既存のシステムを「Agent-Ready」にする作業は、決して巨大なシステム改修ではありませんでした。

1. **AI自身にプロジェクトの規約（Skills）を書かせる**
2. **AI自身に検証用のガードレール（MCPツール）を作らせる**

人間と対話しながらこの2つをリポジトリ内に配置するだけで、Vibe Codingのスピード感を維持したまま、チーム開発にも耐えうる堅牢な「エージェント共生型」の開発基盤が完成します。

アーキテクチャの保護だけでなく、テストデータの自動生成やドメイン固有のビジネスルール検証など、AIが「より自律的に、より安全に」働けるツール（MCP）を拡充していくことで、Agent-Readyな開発保守環境は際限なく強力になっていきます。
